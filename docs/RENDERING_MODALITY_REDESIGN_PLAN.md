# Rendering Modality Redesign Plan (Mesh / Graph / Point Cloud as First-Class Approaches)

## 1) Problem Statement and Goal

The current runtime has strong per-feature rendering paths (mesh-centric plus point cloud pass extensions), but modality selection is not modeled as an explicit first-class contract of the renderer. We need a unified architecture where:

1. **Mesh rendering**, **graph rendering**, and **point cloud rendering** are explicit top-level rendering approaches.
2. Each approach can expose multiple **render modes** (e.g., points as flat disks, surfels, EWA splats, Potree-style screen-space point sizing).
3. Approach + mode selection is user-toggleable at runtime (per-view and per-entity-class), while preserving GPU-driven batching and predictable frame graph behavior.
4. The design remains robust for degenerate geometry, bindless-friendly, and consistent with the engine’s CPU task graph + GPU frame graph split.

---

## 2) Theoretical Model (Unifying “Approach” vs “Mode”)

Define a rendered sample set $\mathcal{S}$ for a scene view $v$:

$$
\mathcal{S}(v) = \mathcal{S}_{mesh}(v) \cup \mathcal{S}_{graph}(v) \cup \mathcal{S}_{point}(v)
$$

Each subset is generated by an **approach** $a \in \{mesh, graph, point\}$ and parameterized by a **mode** $m \in \mathcal{M}_a$.

We model visibility and shading output as:

$$
I(p) = \sum_{a}\sum_{m \in \mathcal{M}_a} \chi_{a,m}(v) \cdot \mathcal{R}_{a,m}(p; \Theta_{a,m}, \mathcal{G}_a)
$$

- $\chi_{a,m}(v) \in \{0,1\}$ is the toggle gate (view/feature config).
- $\Theta_{a,m}$ are mode parameters (radius, attenuation, BRDF variant, splat bandwidth, etc.).
- $\mathcal{G}_a$ is geometry payload for that approach.

For point splatting families we use a common kernel form:

$$
w(\mathbf{x}) = \exp\left(-\frac{1}{2}(\mathbf{x}-\mu)^T \Sigma^{-1} (\mathbf{x}-\mu)\right)
$$

- **Flat disc**: binary support in screen disc ($\Sigma$ replaced by radius test).
- **Surfel**: oriented support from normal/tangent frame.
- **EWA**: perspective-correct elliptical covariance in screen space.
- **Potree-style**: adaptive radius from projected spacing / screen error metric.

Degenerate robustness (mandatory):
- If normal magnitude $\|n\| < \epsilon$, fall back to camera-facing kernel.
- If projected covariance is ill-conditioned ($\kappa(\Sigma) > \tau$), clamp eigenvalues or fall back isotropic.
- For graph edges with zero-length segments, collapse to point primitive path.

Complexity targets:
- CPU approach binning: $O(N)$ over renderables.
- GPU mode bucketing via prefix/compaction: $O(N)$ expected, linear memory traffic.
- No per-draw CPU loops over individual objects after packet build.

## 3) Architectural Placement in the 3-Fold Task System

## 3.1 CPU Task Graph (frame preparation)

Add a **Render Approach Classifier Task**:

- Input: visible render proxies.
- Output: `ApproachPacketBuffer` (SoA) partitioned by approach and mode.

Add a **Mode Policy Resolve Task**:

- Resolves effective mode from stacked policy levels:
  1. Global defaults
  2. View override
  3. Entity tag / component override
  4. Debug forced mode

No locks in hot path:
- Use per-worker local bins then merge via prefix offsets.

## 3.2 GPU Frame Graph (execution)

Frame graph introduces first-class passes:

1. `MeshApproachPass`
2. `GraphApproachPass`
3. `PointApproachPass`

Each pass internally executes mode subpipelines via indirect draws/dispatches.

Optional async compute stage:
- `PointModePreprocessCompute` for splat parameter conditioning (covariance regularization, LOD radius).

## 3.3 Async Streaming Graph

Streaming responsibilities:
- Build/refresh point hierarchical metadata (spacing, local density, node LOD ranges) for Potree-like mode.
- Precompute graph edge strip adjacency / polyline chunking for high-quality line rendering.

## 4) Data-Oriented Design (SoA Contracts)

## 4.1 Core packet schema

```cpp
struct ApproachPacketSoA {
    // identity
    uint32_t* entityId;
    uint16_t* approach;   // Mesh=0, Graph=1, Point=2
    uint16_t* mode;       // approach-local mode id

    // draw ranges / geometry handles
    uint32_t* geometryHandle;
    uint32_t* materialHandle;
    uint32_t* transformHandle;

    // mode params index into bindless param table
    uint32_t* modeParamIndex;
};
```

Rules:
- All hot iteration data in SoA.
- `StrongHandle<T>` / generational indices only; no owning pointers.
- Per-frame transient storage from linear allocator.

## 4.2 Mode parameter table

Use a packed, bindless-addressable table:

```cpp
struct PointModeParams {
    float baseRadiusPx;
    float ewaBandwidth;
    float normalWeight;
    float densityScale;
    uint32_t flags; // bitmask for degenerate fallback behavior
};
```

Equivalent typed blocks for mesh and graph modes.

## 4.3 Geometry abstraction

Promote a unified descriptor:

```cpp
enum class GeometryApproach : uint8_t { Mesh, Graph, PointCloud };

struct GeometryViewDescriptor {
    GeometryApproach approach;
    BufferDeviceAddress positions;
    BufferDeviceAddress normals;
    BufferDeviceAddress indicesOrEdges;
    BufferDeviceAddress aux;
    uint32_t count;
    uint32_t layoutFlags;
};
```

This keeps mesh/graph/point as peer entities in upload + draw code.

## 5.2 Graph approach (`GraphApproachPass`)

Modes (initial):
- `LineList`
- `TubeImpostor`
- `NodeDiscs`
- `EdgeHeatmap`

## 5.3 Point approach (`PointApproachPass`)

Modes (initial):
- `FlatDisc`
- `Surfel`
- `EWA`
- `PotreeAdaptive`
- `GaussianSplat` (existing mode retained as optional)

PotreeAdaptive core policy:

$$
r_{px} = \mathrm{clamp}(\alpha \cdot \frac{s_{world}}{z} f_y, r_{min}, r_{max})
$$

where $s_{world}$ is local spacing estimate, $z$ view depth, $f_y$ focal scale.

## 6) API and Module Surface Changes

## 6.1 Runtime-facing config

Add a dedicated runtime config object:

```cpp
struct RenderApproachConfig {
    bool enableMesh = true;
    bool enableGraph = true;
    bool enablePointCloud = true;

    MeshMode defaultMeshMode;
    GraphMode defaultGraphMode;
    PointMode defaultPointMode;
};
```

Per-view override:

```cpp
struct ViewRenderApproachOverride {
    std::optional<MeshMode> meshMode;
    std::optional<GraphMode> graphMode;
    std::optional<PointMode> pointMode;
    uint32_t disableMask; // bitmask by approach
};
```

## 6.2 Render graph builder integration

Replace feature-specific ad hoc toggles with explicit approach scheduling:

- `ScheduleApproachPasses(const RenderApproachConfig&, const ViewRenderApproachOverride&)`
- Adds only enabled approach passes.

## 6.3 Editor UX contract

Inspector hierarchy:
- `Rendering > Approaches > Mesh / Graph / PointCloud` toggles.
- Nested mode dropdown + per-mode parameters.
- Optional “lock mode across selected entities”.

## 7) Migration Plan (Incremental, Safe)

### Phase 0: Introduce enums + config plumbing (no visual change)
- Add approach/mode enums and packet fields.
- Route existing mesh + point behavior through compatibility mapping.

### Phase 1: First-class pass split
- Extract/rename current pass logic into approach passes.
- Keep old shader code paths wrapped by new mode dispatch.

### Phase 2: Graph approach formalization
- Build graph mode pipeline and indirect packet path.
- Add graph-specific mode params and UI.

### Phase 3: Point mode expansion
- Add PotreeAdaptive mode.
- Consolidate existing flat/surfel/EWA/Gaussian under unified mode table.

### Phase 4: Cleanup and deprecation
- Remove legacy render toggles.
- Delete compatibility shims once stable.

Rollback strategy:
- Runtime feature flag `RenderApproachV2` gates full path.
- One-click fallback to legacy scheduling until parity is validated.

## 8) Robustness and Numerical Safeguards

1. **Normals**: renormalize with epsilon guard; fallback to view-facing basis.
2. **EWA covariance**: SPD enforcement by eigenvalue floor $\lambda_i \leftarrow \max(\lambda_i, \lambda_{min})$.
3. **Potree radius**: clamp to avoid popping and overdraw explosion.
4. **Graph tubes**: Frenet-frame instability on near-collinear segments handled by parallel transport frame fallback.
5. **Depth conflicts**: mode-specific depth bias policy to avoid edge/point z-fighting against meshes.

## 9) Performance Plan and Budgets

CPU (<2ms target contribution):
- Classification + packet build: <0.25 ms for 200k visible primitives.
- Mode policy resolve: <0.10 ms.

GPU:
- Use one indirect command buffer per approach.
- Prefer persistent pipelines and descriptor set layouts; mode selection via push constants + mode param index.
- Use occupancy-friendly workgroup sizing for preprocess compute (point modes).

Telemetry to add:
- `Render/ApproachClassifier`
- `Render/MeshApproachPass`
- `Render/GraphApproachPass`
- `Render/PointApproachPass`
- `Render/PointPreprocessCompute`

## 10) Validation Plan

## 10.1 Correctness
- Golden image comparisons per approach/mode.
- Degenerate fixture scenes:
  - zero normals
  - coplanar/edge-on surfels
  - zero-length graph edges
  - extremely sparse/dense point clouds

## 10.2 Performance
- Automated scene sweep recording CPU packet build + GPU pass time.
- Threshold tests fail CI if regression >10% against baseline.

## 10.3 UX
- Verify toggling any approach/mode is frame-stable (no flicker, no stale buffers).
- Verify coexistence of mixed modes across entities in same frame.
