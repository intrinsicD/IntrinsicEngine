# GRAPHICS-031 — Default debug surface material and missing-material fallback (planning)

## Goal
Lock down the design for a deterministic default/debug surface material plus an
explicit, testable missing-material fallback policy that GRAPHICS-032's minimal
surface pass can rely on, before any shader, registry, or runtime code is
written. This is a planning task; no shaders, no registry changes, no runtime
behavior land here.

## Non-goals
- No implementation, no shader source added, no `MaterialRegistry` changes, no pipeline-state additions in this slice.
- No surface or present pass command bodies (GRAPHICS-032).
- No material registry contract changes from GRAPHICS-006/006Q beyond what the default debug material specifies as a consumer.
- No textured / PBR / clustered / lit material work; default is intentionally untextured/unlit.
- No shader hot reload or asset-backed material loading (GRAPHICS-023 / GRAPHICS-015 / GRAPHICS-034).
- No editor UI or live-registry access from graphics.

## Context
- Owner layer: `graphics` (registry + pipeline state + shader assets); runtime configures default-slot population on extraction. Final material lives in graphics-owned material/pipeline registry from GRAPHICS-006.
- Geometry alone is insufficient for a surface draw; without a default material, GRAPHICS-032's pass body must soft-skip on missing material — exactly the failure mode the 2026-05-08 review forbids.
- The 2026-05-08 review (section "Exact missing pieces / 4") requires a deterministic default with explicit, testable fallback when material assets are absent.
- `src/graphics/renderer/Graphics.GpuWorld.cppm` already exposes `SetInstanceMaterialSlot`; this task decides what slot value the default occupies.
- `src/graphics/renderer/Graphics.Material.cppm:39` already exposes `inline constexpr std::uint32_t kDefaultMaterialSlotIndex = 0u;` and `MaterialFlags::Unlit` (line 126); GRAPHICS-031 reuses both rather than introducing parallel constants.
- `src/graphics/renderer/Graphics.MaterialSystem.cppm:81` already publishes `MaterialSystemDiagnostics::FallbackSlotResolveCount` for stale-handle resolution to slot 0 (incremented in `Graphics.MaterialSystem.cpp:604`); the new GRAPHICS-031 counters are appended to that same struct rather than introducing a parallel `MaterialDiagnostics` snapshot.
- `assets/shaders/debug_surface.vert` exists today and is a *transient* debug-overlay shader (BDA-pointer push constants, no scene-table/MaterialBuffer access). The GRAPHICS-031 default debug surface is a *retained scene-table consumer*; its shader pair is distinct from the transient overlay and is named accordingly (Decision 3).
- `src/legacy/Graphics` debug material code is reference-only and must not be copied.

## Recorded design decisions

Each decision below is locked for downstream implementation children. Trade-offs
are recorded so reviewers can see what was rejected and why.

1. **Material identity and slot.**
   - Decision: option (a). The default debug surface material occupies the existing graphics-owned compile-time slot constant `Extrinsic::Graphics::kDefaultMaterialSlotIndex = 0u` (declared at `src/graphics/renderer/Graphics.Material.cppm:39`). `MaterialSystem::Initialize()` continues to pre-populate slot 0 at construction and must never free it; the only change is *what params/flags are written into slot 0* (Decision 4) and the registered instance name (Decision 2). `MaterialSystem::RebuildGpuResources(device, bufferMgr)` (declared at `Graphics.MaterialSystem.cppm:116`) repopulates slot 0 with byte-identical params after a non-operational → operational device transition; the slot index never changes across rebuilds. The slot value is a compile-time constant for tests; `MaterialLayoutContract::DefaultSlot` (`Graphics.Material.cppm:56`) continues to report `kDefaultMaterialSlotIndex` and is the canonical CPU-readable export of the value.
   - Rejected alternatives: (i) adding a *second* well-known slot (`kDefaultDebugSurfaceMaterialSlotIndex = 1u`) alongside the existing slot 0 PBR default — rejected because it splits "default material" semantics across two slots, forces `FallbackSlotResolveCount` (today routed to slot 0) and the new missing-material fallback to disagree on which slot the runtime/snapshot consumer points at, and adds a graphics-module surface constant that is purely redundant with slot 0 once `docs/architecture/graphics.md:529` already records "the GRAPHICS-031 default debug surface material is unlit by contract"; (ii) a runtime-allocated slot whose value is published via diagnostics — rejected because contract tests would have to read that diagnostic before every assertion and could not pin the slot value at compile time, defeating the GRAPHICS-032 testability goal.

2. **Material name and namespace.**
   - Decision: the canonical name attached to slot 0 is `"Material.DefaultDebugSurface"`. The string lives under graphics-owned material naming (no `Asset.` / `Runtime.` / `ECS.` prefix). It is recorded in `MaterialTypeDesc::Name` only when a new dedicated `MaterialTypeDesc{ .Name = "Material.DefaultDebugSurface", .CustomParams = {} }` is registered (Decision 4); the string is also stable for diagnostics, log breadcrumbs, and editor labels. The display label policy is "exactly the canonical name; no localization, no abbreviation, no per-build suffix". `MaterialSystem::RegisterType()` already rejects duplicate names by returning an invalid handle and incrementing `DuplicateTypeNameCount` (`Graphics.MaterialSystem.cppm:78`); this prohibition is therefore enforced by the existing registration path with no new code, and any user-authored material whose `MaterialTypeDesc::Name` collides with `"Material.DefaultDebugSurface"` fails registration deterministically. The forbidden-collision rule is documented adjacent to `kMaterialTypeID_StandardPBR` / `kMaterialTypeID_SciVis` in `Graphics.Material.cppm`.
   - Rejected alternatives: (i) attaching the string to the slot index alone (no `MaterialTypeDesc`) — rejected because there is no slot-name surface today; building one adds engine surface that this slice forbids, and reusing `MaterialTypeDesc::Name` is layer-clean; (ii) reusing `kMaterialTypeID_StandardPBR` (slot 0 today branches to StandardPBR shading) — rejected because the default debug surface is unlit by contract (Decision 4) and must report a different `MaterialTypeID` so the fragment shader can branch on it without inspecting `MaterialFlags::Unlit` per-fragment; the new constant is `kMaterialTypeID_DefaultDebugSurface` (Decision 6).

3. **Shader pair.**
   - Decision: source language is GLSL (matches the existing `assets/shaders/forward/*.vert/.frag`, `surface.vert/frag`, `debug_surface.vert`, etc., compiled to SPIR-V at build time per the existing `assets/shaders/` toolchain). The shader files are:
     - `assets/shaders/forward/default_debug_surface.vert` — minimal vertex transform: read object-space position from BDA, transform to clip via the canonical `gl_Position = camera.proj * camera.view * inst.Model * vec4(inPos, 1.0)` per the existing `surface.vert` pattern, optionally read packed RGBA8 vertex color from BDA, and forward the unpacked color to the fragment stage. No normal transform, no tangent-space basis, no texcoord output, no shadow-cascade lookups, no per-instance visibility remap branches.
     - `assets/shaders/forward/default_debug_surface.frag` — flat-shaded output: write `outColor = mix(material.BaseColorFactor, fragVertexColor, vertexColorMixFactor)` to the single forward color attachment, where `vertexColorMixFactor` is a compile-time constant in the shader (1.0 when `PtrVertexAttr != 0`, otherwise 0.0). No lighting, no shadowing, no texture sampling, no `discard`, no alpha-blend.
   - Build wiring: shaders are compiled at build time by the existing `assets/shaders/` glslang/glslc step; no runtime shader compilation in this slice. The compiled SPIR-V binaries land under the same generated artifact path as the other shaders in `assets/shaders/forward/`.
   - Rejected alternatives: (i) HLSL — rejected because the entire `assets/shaders/` tree is GLSL today and a parallel HLSL toolchain is not in scope; (ii) reusing `assets/shaders/debug_surface.vert/frag` (the transient debug-overlay shader) — rejected because that pair reads a transient BDA push-constant pointer, not the scene-table SSBO + Material SSBO, and re-purposing it would either fork its uses or drop GpuWorld renderable support; (iii) reusing `assets/shaders/surface.vert/frag` — rejected because that shader does PBR lighting, shadow-cascade sampling, and per-vertex/per-face/centroid/Voronoi attributes the default debug surface explicitly excludes (Decision 11 performance bounds).

4. **Pipeline state.**
   - Decision (locked vector):
     - `CullMode = Back` (cull back faces); the default debug surface is closed-mesh-friendly. Double-sided debug variants are out of scope (Decision 10).
     - `DepthTest = enabled`, `DepthCompareOp = Less`, `DepthWriteEnabled = true`. The pass either clears `SceneDepth` to 1.0 (when run without depth prepass) or loads `SceneDepth` and uses `CompareOp = Equal` with depth-write off (when GRAPHICS-032 routes through `Pass.DepthPrepass` first); the pipeline variant matches the GRAPHICS-008Q renderpass-attachment ownership rule recorded in `docs/architecture/rendering-three-pass.md` ("Renderpass attachment ownership for the depth, surface, and G-buffer paths is fixed").
     - `BlendEnabled = false` on the single forward color attachment.
     - `PolygonMode = Fill`.
     - `PrimitiveTopology = TriangleList`.
     - `MSAA samples = 1` (the canonical surface lane is 1× MSAA today; GRAPHICS-035..058 explicitly does not unblock).
     - `DynamicState = { Viewport, Scissor }` only — no dynamic blend or depth state to keep pipeline-state churn at zero.
     - `Bindings = GpuScenePushConstants` (scene-table BDA + per-instance index ranges) only; no per-material descriptor expansion (Decision 6).
   - The bucket lane is the existing `SurfaceOpaque` cull bucket (`docs/architecture/rendering-three-pass.md:153`); the GRAPHICS-007Q `VisibilityMask`/`Layer` policy is consumed without redefinition. Selection participation is opt-in per the existing `GpuRender_Selectable` flag and follows the eight-bucket cull contract unchanged.
   - Rejected alternatives: (i) `CullMode = None` (double-sided) — rejected because it costs ~2× fragment work for a debug surface and the legitimate double-sided use case is the to-be-defined wireframe variant in Decision 10, not the default; (ii) `BlendEnabled = true` with a 50% alpha — rejected because alpha-blend collides with the fixed `Pass.Forward.Surface` attachment policy and breaks deterministic pixel-readback assertions (Decision 9 visibility guarantee); (iii) introducing a `DefaultDebugSurface`-specific `CompareOp = Always` to bypass depth — rejected because it produces non-deterministic pixel ordering for overlapping triangles and breaks the GRAPHICS-032 minimal-present visibility test.

5. **Vertex format.**
   - Decision: position `vec3` (12 B, scalar-packed) + per-vertex packed color `uint32` (RGBA8, 4 B, unpacked via `unpackUnorm4x8`). Total per-vertex stride: 16 B (when both buffers are present); the layout uses two separate BDA buffers (positions SSBO + colors SSBO), not interleaved, to match the existing `surface.vert` SoA pattern. Color buffer is optional: when `PtrVertexAttr == 0` the fragment shader uses `material.BaseColorFactor` directly (Decision 3 `vertexColorMixFactor = 0`).
   - Interaction with GRAPHICS-030's procedural-source packer: the `Triangle` packer in `Extrinsic.Runtime.ProceduralGeometryPacker` (recorded by GRAPHICS-030 Decision 6 / 13, Impl-A) emits exactly this layout — three `vec3` positions in the position buffer and three `uint32` packed RGBA8 colors in the color buffer. The `Triangle` packer's color values are deterministic and recorded by GRAPHICS-030-Impl-A (planned: red / green / blue corner colors so the visibility guarantee in Decision 9 includes color-corner ordering). No additional vertex attributes (normals, tangents, UVs, second UVs, weights) are required by the default debug surface; if a future packer wants to emit them, they are simply ignored by `default_debug_surface.vert`.
   - Rejected alternatives: (i) interleaved `Vertex { vec3 pos; uint32 color; }` — rejected because the existing `surface.vert` pattern is SoA + BDA, and interleaving would force GRAPHICS-030-Impl-A's packer to allocate a second layout family solely for this shader; (ii) `vec4` color (full float per channel, 16 B) — rejected because `unpackUnorm4x8` on a `uint32` matches the existing `debug_surface.vert` / `surface.vert` color path and keeps the stride at 16 B per vertex; (iii) requiring `vec3` normal — rejected because the default debug surface is unlit (no normal consumer in the fragment shader); demanding normals from the procedural packer doubles its scratch allocation for no observable shading effect.

6. **Descriptor / push-constant layout.**
   - Decision: the default debug surface uses **only** the canonical scene-table push constant defined for the depth/surface/G-buffer family (`docs/architecture/rendering-three-pass.md:174`): `GpuScenePushConstants` carrying `SceneTableBDA` (`uint64_t`) plus the per-draw indirection consumed by `DrawIndexedIndirectCount` (`firstInstance` from the `SurfaceOpaque` bucket) and the existing per-batch `VisibilityBase`/`PtrPositions`/`PtrVertexAttr` BDA pointers from `surface.vert`'s push-constant block (the per-batch pointer set is reused as-is so the pipeline layout matches the existing surface lane). No per-material descriptor *set* is introduced. The fragment shader reads `MaterialBuffer.Materials[inst.MaterialSlot]` from `set = 3, binding = 0` — the existing material SSBO already in the scene-table contract (`docs/architecture/rendering-three-pass.md:131`). A new well-known constant `inline constexpr std::uint32_t kMaterialTypeID_DefaultDebugSurface = 2u;` is recorded in `Graphics.Material.cppm` adjacent to `kMaterialTypeID_StandardPBR = 0u` / `kMaterialTypeID_SciVis = 1u` so the shader can branch by `MaterialTypeID` without inspecting `MaterialFlags`. Total push-constant size remains ≤ 128 B (no growth).
   - Rejected alternatives: (i) introducing a dedicated per-material descriptor set for the default debug surface — rejected because every existing forward/surface pass already routes through `MaterialBuffer` at `set = 3, binding = 0`; an extra set duplicates state and breaks the GRAPHICS-013AQ descriptor-bind seam; (ii) packing `BaseColorFactor` directly into push constants instead of reading the material SSBO — rejected because it bypasses `MaterialSystem::SyncGpuBuffer()`, prevents editor-time tweaks, and forces the fallback substitution path (Decision 7) to push different constants per-draw rather than substituting an SSBO slot index; (iii) adding a `kMaterialTypeID_DefaultDebugSurface` value greater than 2 to leave room for future built-in IDs — rejected because the registry order is the source of truth for IDs, and reserving gaps invites drift between the constant and the actual registration order.

7. **Fallback selection ownership.**
   - Decision: option (b) is canonical; option (a) is permitted as a runtime authoring shorthand only. **(b)** Graphics applies the substitution at snapshot consumption: when the renderer copies a `RuntimeRenderSnapshotBatch` into `RenderWorld` and a renderable's resolved material slot is missing or invalid, the renderer substitutes `kDefaultMaterialSlotIndex` *and* increments the appropriate counter from Decision 8. Substitution happens during the same span-copy that already enforces `InvalidSnapshotRecordCount` rules on `Graphics.Renderer.cpp:355..415` (so the substitution lives next to the existing invalid-record drain, not in a new path). Runtime stays agnostic of graphics-side slot identity. **(a)** Optionally, a renderable may *explicitly* request the default debug surface by setting its CPU material description to the sentinel "no material" value (today: a default-constructed `Graphics::Components::RenderSurface` with no asset-backed material lease and no SciVis instance); in that case `Runtime.RenderExtraction` writes the runtime-side resolved-material-slot field as the sentinel "unset" and the renderer's path (b) substitution applies. There is no graphics-side knowledge of runtime sentinels and no runtime-side knowledge of the slot integer. The two paths converge at the same renderer substitution point; only the counter differs (`MissingMaterialFallbackCount` for unset/sentinel, `InvalidMaterialSlotCount` for out-of-range/stale slot integers).
   - Rejected alternatives: (i) option (a) alone (runtime writes `kDefaultMaterialSlotIndex` directly) — rejected because runtime would import graphics-internal slot integers and any future renumbering would break the runtime side without graphics knowing; (ii) silent skip when material is missing — rejected by the `Forbidden changes` clause "No silent-skip fallback paths" and by the GRAPHICS-032 minimal-pass requirement; (iii) erroring out at extraction time — rejected because debug authoring (entity with geometry + no material) is a legitimate steady-state, not a contract violation.

8. **Diagnostic counters.**
   - Decision: the three counters land as additive fields on the existing `MaterialSystemDiagnostics` struct (`Graphics.MaterialSystem.cppm:75-92`), appended after the existing `Capacity` field so existing test assertions continue to compile. Names exactly:
     - `MissingMaterialFallbackCount` — incremented when the renderer copy step substitutes `kDefaultMaterialSlotIndex` because the snapshot record carried the sentinel "unset" material (Decision 7 path b applied to path a's authoring shorthand).
     - `InvalidMaterialSlotCount` — incremented when the renderer copy step substitutes `kDefaultMaterialSlotIndex` because the snapshot record carried an out-of-range or stale slot integer (slot ≥ live capacity, slot referenced a destroyed instance).
     - `DefaultDebugSurfaceUses` — incremented once per resolved per-frame renderable record whose final `MaterialSlot == kDefaultMaterialSlotIndex` after substitution (so the counter reports total *uses* of the default, not just *fallback applications*; the difference between `DefaultDebugSurfaceUses` and `(MissingMaterialFallbackCount + InvalidMaterialSlotCount)` is the count of explicit-default authoring).
   - Surfacing path: counters are exposed through the existing `MaterialSystem::GetDiagnostics()` accessor (`Graphics.MaterialSystem.cppm:216`) and reset at the same cadence as the existing diagnostics (per-frame zero on `BeginFrame()` for monotonically-per-frame counters, mirroring the `InvalidSnapshotRecordCount` reset at `Graphics.Renderer.cpp:355` and `Graphics.Renderer.cpp:907`). Forbid silent-skip: every fallback path increments exactly one counter from this set. The GRAPHICS-002 `InvalidSnapshotRecordCount` and GRAPHICS-012Q `Picking.Readback` patterns are mirrored — counters are CPU-side, deterministic, and contract-testable without Vulkan.
   - Rejected alternatives: (i) introducing a new `MaterialDiagnostics` snapshot struct alongside the existing `MaterialSystemDiagnostics` — rejected because the existing struct already aggregates material-related fallback counters (`FallbackSlotResolveCount`, `IncompatibleLayoutCount`, `InvalidCreateTypeCount`, `CapacityFailureCount`) and adding three more fields is the minimal surface change; (ii) embedding the counters on `RenderWorld` next to `InvalidSnapshotRecordCount` — rejected because the counters describe a *substitution outcome*, not a *rejection*, and the sibling `RenderWorld::InvalidSnapshotRecordCount` is reserved for records that are dropped, not records that are rewritten; (iii) collapsing `MissingMaterialFallbackCount` and `InvalidMaterialSlotCount` into a single counter — rejected because the two error modes have different root causes (authoring gap vs. lifetime/race) and downstream diagnostics needs to distinguish them.

9. **Visibility guarantee.**
   - Decision: "visible" means a deterministic, non-black RGB color rendered into the forward color attachment such that a pixel-readback or CPU-mock command-stream test can assert it. The locked color is `BaseColorFactor = { 0.55, 0.20, 0.85, 1.0 }` (a saturated magenta-ish purple, RGB=`(140, 51, 217)` after × 255 rounding). Rationale: distinct from the canonical clear color (black or `{0,0,0,1}`), distinct from the existing `LightSystem` directional fallback color, distinct from the `surface.frag` PBR opaque-white default, distinct from `R8G8B8A8_UNORM` debug-view fallback gradients, and intentionally chosen so a "missing material" state is recognizable in screenshots without reading diagnostics. Per-vertex color (Decision 5) overrides this when the procedural packer supplies vertex colors (e.g. GRAPHICS-030's red/green/blue triangle), but the alpha is always `1.0` and `BaseColorFactor` is the deterministic value when no vertex colors are present. The pixel-readback test (when GPU is opt-in, Decision 12) asserts the center pixel of the rendered triangle equals the unpacked vertex color (or `BaseColorFactor` if vertex colors are absent) within a 1-LSB tolerance per channel.
   - Rejected alternatives: (i) bright magenta `{1,0,1,1}` — rejected because it visually collides with the canonical "missing texture" magenta many engines use, and the GRAPHICS-031 default debug surface should be distinguishable from texture residency failure (which has its own fallback texture in `Graphics.GpuAssetCache`); (ii) procedural per-frame color (e.g. animated hue) — rejected because non-deterministic colors break pixel-readback assertions; (iii) opaque white `{1,1,1,1}` — rejected because it's identical to the previous slot 0 PBR opaque-white default and prevents tests from distinguishing "default debug surface rendered" from "lit PBR result happens to be white".

10. **Extensibility surface.**
    - Decision: follow-up debug materials (line, point, wireframe, normals, UVs, depth, instance-id) attach as additional `MaterialTypeDesc` registrations and additional well-known slot constants in `Graphics.Material.cppm`, not as forks of `MaterialSystem` or new descriptor sets. Each future debug material shares the same descriptor layout family (no per-material descriptor set, scene-table BDA + per-instance index, `MaterialBuffer` SSBO at `set = 3, binding = 0`); only the registered `MaterialTypeID`, the registered `MaterialTypeDesc::Name`, and the shader pair differ. Naming family: `Material.DefaultDebug<Variant>` (`Material.DefaultDebugSurface`, `Material.DefaultDebugWireframe`, `Material.DefaultDebugLine`, `Material.DefaultDebugPoint`, `Material.DefaultDebugNormals`, `Material.DefaultDebugUVs`, `Material.DefaultDebugDepth`, `Material.DefaultDebugInstanceId`). Slot-constant family: `kDefaultDebug<Variant>MaterialSlotIndex`, allocated immediately after `kDefaultMaterialSlotIndex` at `MaterialSystem::Initialize()`. Each variant's substitution counter is named `DefaultDebug<Variant>Uses` on `MaterialSystemDiagnostics` (Decision 8 pattern).
    - In-scope for this slice: exactly one variant (`DefaultDebugSurface` at slot 0). All other variants are identified but **not** opened here; they require a follow-up planning slice that records per-variant decisions and a separate implementation child.
    - Rejected alternatives: (i) baking variants behind shader `#define`s on a single registered material — rejected because branching by define forks the SPIR-V at build time and prevents per-instance variant selection; (ii) storing the variant in a `MaterialFlags` bit — rejected because flags are 1-bit boolean state (Unlit, AlphaMask, etc.) and the variant is a multi-valued enum that maps better onto `MaterialTypeID`; (iii) opening a single GRAPHICS-031-Impl-D child for *all* variants — rejected because each variant needs its own visibility guarantee, vertex format, and pipeline state.

11. **Performance bounds.**
    - Decision (locked):
      - **Shader instruction-count budget.** `default_debug_surface.vert` ≤ 32 SPIR-V instructions (estimate: 1 BDA load × 2, 1 mat4×vec4 transform via two mat4 loads, 1 optional `unpackUnorm4x8`, 1 conditional vertex-color branch, ~4 input-output forwarding instructions). `default_debug_surface.frag` ≤ 16 SPIR-V instructions (1 SSBO load on `MaterialBuffer.Materials[slot]`, 1 mix or select on vertex color, 1 store to color attachment).
      - **Forbidden in the shader pair:** loops, texture samples, sampler bindings, push-constant growth beyond the existing `surface.vert` push-constant block size, dynamic indexing into multi-material buffers other than the canonical `MaterialBuffer.Materials[inst.MaterialSlot]`.
      - **Pipeline creation.** Exactly one graphics pipeline created at renderer init (`MaterialSystem::Initialize()` time, alongside slot 0 population). Zero per-frame state churn; the pipeline is owned for the renderer lifetime and only torn down at `Shutdown()`. No pipeline rebuild on `RebuildGpuResources()` unless the device's bindless heap or scene-table descriptor set layout actually changed (the existing rebuild rule from `Graphics.MaterialSystem.cppm:115`).
      - **Material slot SSBO traffic.** Slot 0's `GpuMaterialSlot` is uploaded exactly once at `MaterialSystem::Initialize()` and republished on `RebuildGpuResources()`; it is never marked dirty per-frame in the default contract (no animation, no editor tweak, no runtime mutation).
      - **Runtime substitution cost.** The Decision 7 path-(b) substitution runs in O(1) per renderable record during the snapshot copy step that already iterates `RenderableSnapshot` records on `Graphics.Renderer.cpp:355..415`; no extra pass, no extra map lookup, no per-frame allocation.
    - Rejected alternative: deferring pipeline creation to first-use — rejected because first-use is the GRAPHICS-032 minimal-pass first frame, and any allocation/compile cost there delays the visibility milestone the entire Theme A line targets.

12. **Layering audit.**
    - Decision: GRAPHICS-031 introduces zero new dependency edges. The default debug surface lives entirely inside `src/graphics/renderer/`:
      - Slot constant additions (`kMaterialTypeID_DefaultDebugSurface`, the `DefaultDebugSurfaceUses`/`MissingMaterialFallbackCount`/`InvalidMaterialSlotCount` counter fields, the `"Material.DefaultDebugSurface"` string registration) live in the existing `Extrinsic.Graphics.Material` and `Extrinsic.Graphics.MaterialSystem` modules; no new module is created.
      - The shader pair lives under `assets/shaders/forward/`, the same generated-asset root every other forward shader uses; assets are not a `src/` layer and do not introduce a layering edge.
      - `runtime` does not import the shader, the slot constant, or any pipeline-state internal. Runtime only emits the existing "unset material" sentinel through `Runtime.RenderExtraction` per Decision 7 path (a) shorthand; the renderer-side substitution (Decision 7 path b) is graphics-internal.
      - `ecs` never references the material; canonical ECS components do not store `kDefaultMaterialSlotIndex` per the existing contract in `docs/architecture/graphics.md:14` ("Graphics-owned GPU handles, slots, leases, and backend resource IDs must not be stored in canonical `src/ecs` components").
      - `assets` is untouched: the default material is registry-owned at `MaterialSystem::Initialize()` time and is not asset-loaded. No `Assets::AssetId` ever references the default debug surface; the GRAPHICS-023A/B/C/D asset-generation observation paths short-circuit on `!HasSourceAsset()` for the default just as they do for procedural sources (GRAPHICS-030 Decision 5).
      - `graphics/rhi` is untouched: the new `MaterialTypeID` constant is a CPU-side scalar; the SSBO layout is unchanged; the pipeline state vector (Decision 4) maps onto existing RHI pipeline-creation surfaces.
    - The promoted layering invariants `core ← geometry`, `core ← assets`, `core ← ecs`, `core, asset-IDs, RHI ← graphics`, "no live ECS in graphics", and "no graphics handles in canonical ECS components" all hold without exception.
