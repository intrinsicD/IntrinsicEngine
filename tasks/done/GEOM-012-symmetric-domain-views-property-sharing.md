## Slice plan
- **Slice A — Mesh-backed graph borrow adaptor (this slice, landed; topology-safety fix follow-up landed).** Adds the new `Geometry.DomainViews` module exposing `Geometry::DomainViews::BorrowMeshAsGraphReadOnly(const HalfedgeMesh::Mesh&) -> Graph::Graph`. The factory constructs a `Graph::Graph` that shares the source mesh's vertex/halfedge/edge `PropertySet`s and deleted-vertex/edge counters; `Graph::EnsureProperties` reuses the mesh's existing `v:point`, `v:connectivity`, `h:connectivity`, `v:deleted`, `e:deleted` slots and does not allocate compatibility-copy properties. Face storage (`h:face`, `f:connectivity`, `f:deleted`, `FacesSize()`, `DeletedFaceCount()`) is deliberately excluded from the view: the const-reference parameter is the safety signal that topology mutation through the returned graph (`AddVertex`/`AddEdge`/`DeleteVertex`/`DeleteEdge`/`GarbageCollection`/`Clear`/`SetNextHalfedge`/etc.) is undefined behavior on face-bearing source meshes because those graph methods cannot update face incidence. Graph-domain reads and vertex-position writes are explicitly allowed (positions do not change face incidence). Promotes the `MakeMeshBackedGraphView` test helper out of `Test_ShortestPath.cpp` and updates all callers to use the public API. New `Test_SubmeshViewDomainBorrows.cpp` proves shared property identity, absence of compatibility-copy `*_graph_*` slots, size sharing with face-state isolation (FacesSize/DeletedFaceCount untouched, mesh's `h:face`/`f:connectivity` preserved), mesh→view position-edit visibility, view→mesh position-edit visibility, and the empty-mesh case. A distinct compile-time-checked read-only view type is owned by Slice D; a face-free mutable-borrow factory (if needed by later callsites) is owned by Slice D or a separate follow-up. Defers point-cloud direction (Slices B/C) and hard-copy conversion seams (Slice E).
- **Slice B — Mesh-backed point-cloud borrow adaptor (this slice, landed; deletion-counter and subview correctness follow-ups landed).** Adds `Geometry::DomainViews::BorrowMeshAsCloud(HalfedgeMesh::Mesh&) -> PointCloud::Cloud`. The factory routes through a new `PointCloud::Cloud(PropertySet&)` constructor that shares the source mesh's vertex `PropertySet` while owning the cloud's own deletion counter; cloud-side deletes mark `p:deleted` on the shared `PropertySet` but do **not** increment `mesh.DeletedVertexCount()`, so the mesh's `VertexCount()`/`HasGarbage()` continue to reflect only mesh-side `v:deleted`. The canonical `v:point` slot is reused — no `p:position` compatibility-copy property is allocated. Existing per-vertex attributes on the mesh (for example `v:normal`) are reachable through `Cloud::GetVertexProperty<T>` over the shared `PropertySet`. `PointCloud::Cloud::AddPoint` appends a row to the shared vertex `PropertySet`, so a point added through the returned view becomes a new vertex slot on the source mesh; the new vertex is isolated (no incident halfedges) so face/halfedge/edge state on a face-bearing source mesh is not corrupted. `Cloud::GarbageCollection` on a mesh-backed borrow is documented undefined behavior on face-bearing meshes because it physically reshuffles and resizes vertex slots; topology-aware deletion must still be routed through `Mesh::DeleteVertex`/`Mesh::GarbageCollection`. The pre-existing `Cloud::CreateView` is fixed to clamp and bind against the cloud's bound `m_Vertices`/`m_DeletedVertices` rather than `m_Properties->Vertices`/`m_Properties->DeletedVertices`; for owned clouds this is a no-op rebinding (the references alias the shared `Properties`) and for mesh-borrowed clouds the subview now correctly sees the mesh-backed rows instead of clamping to size 0. Slice B adds eleven new tests in `Test.SubmeshViewDomainBorrows.cpp`: shared `v:point` slot identity, absence of `p:position` compatibility-copy, mesh-side `v:normal` reuse through `Cloud::GetVertexProperty`, mesh→cloud and cloud→mesh position-edit visibility, point-addition propagation onto the source mesh with face state untouched, the empty-mesh case, `Cloud::CreateView` over a mesh-borrowed cloud yields a non-empty subrange tied to mesh-backed rows, `Cloud::CreateView` over an empty-mesh borrow clamps to zero, cloud-side `DeletePoint` leaves the mesh's deletion counter and `IsDeleted` semantics untouched, and a follow-up mesh `GarbageCollection` after a cloud-side delete is a consistent no-op from the mesh's perspective.
- **Slice C — Graph-backed point-cloud borrow adaptor (this slice, landed).** Adds `Geometry::DomainViews::BorrowGraphAsCloud(Graph::Graph&) -> PointCloud::Cloud`, the symmetric companion to Slice B's `BorrowMeshAsCloud`. The factory routes through the existing `PointCloud::Cloud(PropertySet&)` borrow constructor, sharing only the source graph's vertex `PropertySet` (via `Graph::VertexProperties()`) while owning the cloud's own deletion counter; cloud-side deletes mark `p:deleted` on the shared `PropertySet` but do **not** touch the graph's `v:deleted` counter, so the graph's `VertexCount()`/`HasGarbage()` continue to reflect only graph-side semantics. The canonical `v:point` slot is reused — no `p:position` compatibility-copy property is allocated — and existing per-vertex attributes (for example `v:normal`) stay reachable through `Cloud::GetVertexProperty<T>`. Only the vertex `PropertySet` is borrowed: the graph's halfedge/edge storage (`h:connectivity`, `e:deleted`) lives on separate `PropertySet`s the cloud never holds, so it is unreachable through the cloud surface and the graph's `EdgesSize()`/`HalfedgesSize()` are untouched. The graph-domain `v:connectivity` slot, however, lives on the **shared** vertex `PropertySet` and stays physically reachable through generic `PointProperties()` access; the `Cloud` owns no connectivity accessor and never touches it, but mutating or clearing it through the cloud — including via `Cloud::Clear()`/`Cloud::GarbageCollection()` — is documented UB on an edge-bearing source graph (the same topology-mutation boundary as the other borrows). Type-level prevention of reaching graph-domain slots is deferred to Slice D's restricted const-view types. `Cloud::AddPoint` appends a row to the shared vertex `PropertySet`, so a point added through the view becomes a new isolated vertex slot on the source graph; topology-aware deletion must still route through `Graph::DeleteVertex`/`Graph::GarbageCollection`, and `Cloud::GarbageCollection` on a graph-backed borrow is documented UB on edge-bearing graphs because it physically reshuffles vertex slots. Slice C adds ten new tests in `Test.SubmeshViewDomainBorrows.cpp`: shared `v:point` slot identity, absence of `p:position` compatibility-copy, non-exposure of the halfedge/edge `PropertySet`s with graph edge/halfedge sizes untouched, the shared `v:connectivity` slot pinned as a reachable documented-UB boundary, `v:normal` reuse through `Cloud::GetVertexProperty`, graph→cloud and cloud→graph position-edit visibility, point-addition propagation onto the source graph with the new vertex isolated and edge state untouched, the empty-graph case, and cloud-side `DeletePoint` leaving the graph's deletion view/`HasGarbage` untouched.
- **Slice D — Read-only borrow surface (this slice, landed).** Adds `Geometry::DomainViews::ConstMeshBackedGraphView`, `ConstMeshBackedCloudView`, and `ConstGraphBackedCloudView` to `Geometry.DomainViews`. Each wraps the same shared-storage borrow produced by its mutable factory (Slice A/B/C `BorrowMeshAsGraphReadOnly`/`BorrowMeshAsCloud`/`BorrowGraphAsCloud`) — the borrowed container is initialized directly from the factory prvalue so guaranteed copy elision preserves storage sharing rather than deep-copying — but exposes **only** `const`-returning accessors (sizes/counts, validity/deletion, `VertexPosition`/`Position`/`Positions`, attribute-presence flags, `*Properties() const -> ConstPropertySet`, `GetVertexProperty<T> const`) plus an `AsGraph()`/`AsCloud()` accessor returning the underlying container as a `const` reference for interop with const-taking algorithms. There is no `Add*`/`Delete*`/`SetVertexPosition`/`Clear`/`GarbageCollection`/`GetOrAdd*Property`/mutable-element-access member, and the views are non-copyable/non-movable (a borrow bound to its source at construction). Because `ConstGraphBackedCloudView` exposes no mutable property access and no `Clear`/`GarbageCollection`, the shared graph-domain `v:connectivity` slot that Slice C could only document as a UB boundary cannot be mutated/cleared through the read-only view — that boundary is now closed by construction. The const-view constructors take a *mutable* source reference (`Mesh&`/`Graph&`): construction routes through the mutable borrow factories, which lazily materialize the shared `v:point`/`p:deleted`/connectivity columns via `EnsureProperties`/`GetOrAdd` on the source property set, so a genuinely `const` source would require an unsafe `const_cast` into a mutating path (undefined behavior) and is rejected at compile time; the post-construction surface remains read-only. Slice D adds a file-scope compile-time `static_assert` block to `Test.SubmeshViewDomainBorrows.cpp` (detection concepts proven non-vacuous via positive controls on the mutable `Graph::Graph`/`PointCloud::Cloud`, then asserted false for the mutating probes and true for the read probes on each view, plus non-copyable/non-movable assertions) and four runtime tests (compile-time gate marker + shared-storage/live-edit proof for each of the three views). Defers the conversion/move/consume policy review (Slice E).
- **Slice E — Conversion/move/consume policy (this slice, landed).** Reviewed the conversion/move/consume coverage and concluded **no missing pair**: the container copy constructor/assignment is the canonical same-domain hard-copy seam (always allocates a fresh owning `PropertySet` and deep-copies, so copying a borrowed `Graph`/`Cloud` yields an independent owning container that decouples from source mutation and outlives source destruction); `Geometry::Mesh::Conversion` and `Geometry::PointCloud::Conversion` own the cross-domain `To*`/`From*` hard copies; move assignment is the ownership-transfer-into-result seam; and the Slice D `Const*View` types are already non-copyable/non-movable. No new APIs were added. Pinned the policy with a new `Hard-copy, move, and consume policy` subsection in `docs/architecture/geometry.md` and five new tests: three `SubmeshViewDomainBorrows.HardCopyOf*BorrowOwnsIndependentStorage` cases (mesh-backed graph, mesh-backed cloud, graph-backed cloud) proving a copied borrow is mutation-independent and outlives source destruction, plus `MeshConversion.ConvertedHalfedgeMeshOutlivesSourceViaMoveOwnershipTransfer` and `PointCloudConversion.ConvertedCloudOutlivesAndDecouplesFromSource` proving cross-format conversion outputs survive source destruction/mutation and that move-assign transfers ownership into a caller-owned result container. Closes the task at `CPUContracted`.

## Goal
- Define and implement symmetric, no-copy domain views between `Geometry::HalfedgeMesh::Mesh`, `Geometry::Graph::Graph`, and `Geometry::PointCloud::Cloud` where their required property sets are semantically compatible.

## Non-goals
- No broad replacement of existing `Mesh`, `Graph`, or `Cloud` containers.
- No renderer/runtime/ECS/assets/platform/app integration.
- No forced hard-copy conversion as the default for read-only algorithms.
- No semantic algorithm changes beyond view/adaptor contracts and tests.

## Context
- Owning subsystem/layer: `geometry` (`geometry -> core` only).
- Seeded by [`docs/reviews/2026-05-12-src-geometry-gap-analysis.md`](../../../docs/reviews/2026-05-12-src-geometry-gap-analysis.md) and the follow-up domain ownership review on 2026-05-12.
- Current mesh-backed graph usage can share vertex, halfedge, and edge property sets because `HalfedgeMesh::VertexConnectivity` and `HalfedgeMesh::HalfedgeConnectivity` alias the canonical `Geometry::Graph` connectivity records.
- Point-cloud positions are standardized with mesh/graph positions through the canonical `v:point` property. Remaining domain-view work must preserve this invariant and handle any legacy `p:position` data only through explicit compatibility conversions.
- Future graph, point-cloud, and mesh algorithms should accept the minimal domain they need and should not force data copies when a borrowed view is valid.
