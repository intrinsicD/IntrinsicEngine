## Slice plan
- **Slice A — Mesh-backed graph borrow adaptor (this slice, landed; topology-safety fix follow-up landed).** Adds the new `Geometry.DomainViews` module exposing `Geometry::DomainViews::BorrowMeshAsGraphReadOnly(const HalfedgeMesh::Mesh&) -> Graph::Graph`. The factory constructs a `Graph::Graph` that shares the source mesh's vertex/halfedge/edge `PropertySet`s and deleted-vertex/edge counters; `Graph::EnsureProperties` reuses the mesh's existing `v:point`, `v:connectivity`, `h:connectivity`, `v:deleted`, `e:deleted` slots and does not allocate compatibility-copy properties. Face storage (`h:face`, `f:connectivity`, `f:deleted`, `FacesSize()`, `DeletedFaceCount()`) is deliberately excluded from the view: the const-reference parameter is the safety signal that topology mutation through the returned graph (`AddVertex`/`AddEdge`/`DeleteVertex`/`DeleteEdge`/`GarbageCollection`/`Clear`/`SetNextHalfedge`/etc.) is undefined behavior on face-bearing source meshes because those graph methods cannot update face incidence. Graph-domain reads and vertex-position writes are explicitly allowed (positions do not change face incidence). Promotes the `MakeMeshBackedGraphView` test helper out of `Test_ShortestPath.cpp` and updates all callers to use the public API. New `Test_SubmeshViewDomainBorrows.cpp` proves shared property identity, absence of compatibility-copy `*_graph_*` slots, size sharing with face-state isolation (FacesSize/DeletedFaceCount untouched, mesh's `h:face`/`f:connectivity` preserved), mesh→view position-edit visibility, view→mesh position-edit visibility, and the empty-mesh case. A distinct compile-time-checked read-only view type is owned by Slice D; a face-free mutable-borrow factory (if needed by later callsites) is owned by Slice D or a separate follow-up. Defers point-cloud direction (Slices B/C) and hard-copy conversion seams (Slice E).
- **Slice B — Mesh-backed point-cloud borrow adaptor (this slice, landed; deletion-counter and subview correctness follow-ups landed).** Adds `Geometry::DomainViews::BorrowMeshAsCloud(HalfedgeMesh::Mesh&) -> PointCloud::Cloud`. The factory routes through a new `PointCloud::Cloud(PropertySet&)` constructor that shares the source mesh's vertex `PropertySet` while owning the cloud's own deletion counter; cloud-side deletes mark `p:deleted` on the shared `PropertySet` but do **not** increment `mesh.DeletedVertexCount()`, so the mesh's `VertexCount()`/`HasGarbage()` continue to reflect only mesh-side `v:deleted`. The canonical `v:point` slot is reused — no `p:position` compatibility-copy property is allocated. Existing per-vertex attributes on the mesh (for example `v:normal`) are reachable through `Cloud::GetVertexProperty<T>` over the shared `PropertySet`. `PointCloud::Cloud::AddPoint` appends a row to the shared vertex `PropertySet`, so a point added through the returned view becomes a new vertex slot on the source mesh; the new vertex is isolated (no incident halfedges) so face/halfedge/edge state on a face-bearing source mesh is not corrupted. `Cloud::GarbageCollection` on a mesh-backed borrow is documented undefined behavior on face-bearing meshes because it physically reshuffles and resizes vertex slots; topology-aware deletion must still be routed through `Mesh::DeleteVertex`/`Mesh::GarbageCollection`. The pre-existing `Cloud::CreateView` is fixed to clamp and bind against the cloud's bound `m_Vertices`/`m_DeletedVertices` rather than `m_Properties->Vertices`/`m_Properties->DeletedVertices`; for owned clouds this is a no-op rebinding (the references alias the shared `Properties`) and for mesh-borrowed clouds the subview now correctly sees the mesh-backed rows instead of clamping to size 0. Slice B adds eleven new tests in `Test.SubmeshViewDomainBorrows.cpp`: shared `v:point` slot identity, absence of `p:position` compatibility-copy, mesh-side `v:normal` reuse through `Cloud::GetVertexProperty`, mesh→cloud and cloud→mesh position-edit visibility, point-addition propagation onto the source mesh with face state untouched, the empty-mesh case, `Cloud::CreateView` over a mesh-borrowed cloud yields a non-empty subrange tied to mesh-backed rows, `Cloud::CreateView` over an empty-mesh borrow clamps to zero, cloud-side `DeletePoint` leaves the mesh's deletion counter and `IsDeleted` semantics untouched, and a follow-up mesh `GarbageCollection` after a cloud-side delete is a consistent no-op from the mesh's perspective.
- **Slice C — Graph-backed point-cloud borrow adaptor (this slice, landed).** Adds `Geometry::DomainViews::BorrowGraphAsCloud(Graph::Graph&) -> PointCloud::Cloud`, the symmetric companion to Slice B's `BorrowMeshAsCloud`. The factory routes through the existing `PointCloud::Cloud(PropertySet&)` borrow constructor, sharing only the source graph's vertex `PropertySet` (via `Graph::VertexProperties()`) while owning the cloud's own deletion counter; cloud-side deletes mark `p:deleted` on the shared `PropertySet` but do **not** touch the graph's `v:deleted` counter, so the graph's `VertexCount()`/`HasGarbage()` continue to reflect only graph-side semantics. The canonical `v:point` slot is reused — no `p:position` compatibility-copy property is allocated — and existing per-vertex attributes (for example `v:normal`) stay reachable through `Cloud::GetVertexProperty<T>`. Only the vertex `PropertySet` is borrowed: the graph's halfedge/edge storage (`h:connectivity`, `e:deleted`) and the graph-domain `v:connectivity` slot are not exposed through the cloud surface, and the graph's `EdgesSize()`/`HalfedgesSize()` are untouched. `Cloud::AddPoint` appends a row to the shared vertex `PropertySet`, so a point added through the view becomes a new isolated vertex slot on the source graph; topology-aware deletion must still route through `Graph::DeleteVertex`/`Graph::GarbageCollection`, and `Cloud::GarbageCollection` on a graph-backed borrow is documented UB on edge-bearing graphs because it physically reshuffles vertex slots. Slice C adds nine new tests in `Test.SubmeshViewDomainBorrows.cpp`: shared `v:point` slot identity, absence of `p:position` compatibility-copy, non-exposure of edge/halfedge data with graph edge/halfedge sizes untouched, `v:normal` reuse through `Cloud::GetVertexProperty`, graph→cloud and cloud→graph position-edit visibility, point-addition propagation onto the source graph with the new vertex isolated and edge state untouched, the empty-graph case, and cloud-side `DeletePoint` leaving the graph's deletion view/`HasGarbage` untouched.
- **Slice D — Read-only borrow surface.** Introduces `ConstMeshBackedGraphView` / `ConstMeshBackedCloudView` / `ConstGraphBackedCloudView` wrappers whose only accessors are `const`-returning and do not expose `Add*`/`Delete*`/`GetOrAdd*Property`. Compile-time tests assert mutation through the const view is ill-formed. Promotes the mutable-borrow rule from documentation to type.
- **Slice E — Conversion/move/consume policy.** Documents and pins the hard-copy seam (`Geometry::Mesh::Conversion` / `Geometry::PointCloud::Conversion` already cover their pairs) and the explicit ownership-transfer pattern. No new APIs unless review finds a missing pair; closes the task at `CPUContracted`.

## Goal
- Define and implement symmetric, no-copy domain views between `Geometry::HalfedgeMesh::Mesh`, `Geometry::Graph::Graph`, and `Geometry::PointCloud::Cloud` where their required property sets are semantically compatible.

## Non-goals
- No broad replacement of existing `Mesh`, `Graph`, or `Cloud` containers.
- No renderer/runtime/ECS/assets/platform/app integration.
- No forced hard-copy conversion as the default for read-only algorithms.
- No semantic algorithm changes beyond view/adaptor contracts and tests.

## Context
- Owning subsystem/layer: `geometry` (`geometry -> core` only).
- Seeded by [`docs/reviews/2026-05-12-src-geometry-gap-analysis.md`](../../../docs/reviews/2026-05-12-src-geometry-gap-analysis.md) and the follow-up domain ownership review on 2026-05-12.
- Current mesh-backed graph usage can share vertex, halfedge, and edge property sets because `HalfedgeMesh::VertexConnectivity` and `HalfedgeMesh::HalfedgeConnectivity` alias the canonical `Geometry::Graph` connectivity records.
- Point-cloud positions are standardized with mesh/graph positions through the canonical `v:point` property. Remaining domain-view work must preserve this invariant and handle any legacy `p:position` data only through explicit compatibility conversions.
- Future graph, point-cloud, and mesh algorithms should accept the minimal domain they need and should not force data copies when a borrowed view is valid.

- [ ] Hard-copy and move semantics are explicit API choices, not accidental side effects of adaptation.
- [ ] The implementation preserves `geometry -> core` layering.
