The sub-slices below mirror the A/B/C cadence (lifecycle → executor wiring → operational behaviour → push-constant follow-up):

  - **Slice D.1 — landed.** Renderer-owned host-visible `Picking.Readback` buffer lifecycle (no copy, no drain). `NullRenderer` owns `m_PickingReadbackBuffer` (`std::optional<RHI::BufferManager::BufferLease>`) and `m_PickingReadbackBufferSize` (`std::uint64_t`); the buffer is allocated in `InitializeOperationalPassResources(device)` before the selection-ID pipelines, sized `8 * device.GetFramesInFlight()` bytes (one 4-byte `EntityId` word + one 4-byte `EncodedSelectionId` word per in-flight frame slot per `GRAPHICS-012Q`'s `EncodedSelectionId` payload layout) with `HostVisible = true` and `BufferUsage::TransferDst`. The allocator computes the expected size each call and re-uses the existing lease only when `m_PickingReadbackBuffer.has_value() && IsValid() && m_PickingReadbackBufferSize == pickingReadbackBytes`; if the device reports a different frames-in-flight after a swapchain rebuild the lease is dropped and re-created so Slice D.2's `slot * 8` per-frame copy addressing never overruns the allocation. When the frames-in-flight count is unchanged the handle Slice D.2 will pass into the recipe stays byte-identical across rebuilds. This deviates from the reset/republish pattern Slices A/B/C use for pipeline leases because `BufferManager::Create()` does not dedupe by descriptor (each `Create()` yields a fresh handle from `IDevice::CreateBuffer`); the size-guarded lazy pattern mirrors `ShadowSystem::TryAllocateAtlas`'s `AtlasLease.GetHandle().IsValid()` guard, which is the only other host-visible/imported resource the renderer owns across rebuilds. The lease is reset in `Shutdown()` before `m_BufferManager` is torn down so the lease's destructor still observes a live manager (the manager's `Release` path calls `IDevice::DestroyBuffer`). `IRenderer` exposes `GetPickingReadbackBuffer()` / `GetPickingReadbackBufferSize()` returning the underlying `BufferHandle` + size, mirroring the `GetMinimalDebugBackbufferReadbackBuffer()` accessor pattern. Contract tests landed: `PickingReadbackBufferSurvivesOperationalRebuild` asserts the returned handle is valid before and after `RebuildOperationalResources()`, that the size equals `8 * frames-in-flight`, that the handle (Index + Generation) is byte-identical across the rebuild when frames-in-flight is unchanged, and that `Shutdown()` releases the lease so the accessor reverts to an invalid handle / zero size; `PickingReadbackBufferReallocatesWhenFramesInFlightChanges` flips `MockDevice::FramesInFlight` from 2 → 3 across a rebuild and asserts the lease was re-allocated (different handle, size now `8 * 3`), then verifies a second rebuild with the same count keeps the handle stable (`MockDevice::FramesInFlight` is now agent-tunable rather than a fixed `return 2`). Adds no copy command and does not wire the recipe import (D.2 scope); does not drive `SelectionSystem::PublishPickResult` / `PublishNoHit` (D.3 scope); does not touch the outline push constants (D.4 scope). The buffer is intentionally unconsumed at end-of-slice — D.2 imports it into the recipe and records the copy.

  - **Slice D.2 — landed.** Recipe-side import of the renderer-owned buffer + per-frame texture-to-buffer copy. `FrameRecipeImports` now carries `RHI::BufferHandle PickingReadback`; `NullRenderer::ExecuteFrame` plumbs it from `m_PickingReadbackBuffer->GetHandle()` (or an invalid handle when the lease is not yet allocated). `BuildDefaultFrameRecipe` imports the buffer via `graph.ImportBuffer("Picking.Readback", imports.PickingReadback, BufferState::TransferDst, BufferState::HostReadback)` when `pickingActive` is true — the `TransferDst` initial state satisfies the framegraph's imported-write contract (`BufferStateAllowsWrite` requires `ShaderWrite`/`TransferDst` on initial or final state when an imported buffer is written; `HostReadback` alone would fail validation), and the `HostReadback` final state leaves the buffer host-readable for the D.3 drain on `BeginFrame`. The previous `graph.CreateBuffer("Picking.Readback", ...)` transient path is removed; `DescribeDefaultFrameRecipe` now declares `Picking.Readback` with `Imported=true, ImportedWriteAllowed=true` so the recipe-aware validator authorises `PickingPass` as the imported writer (PickingPass's `Writes` list still contains `"Picking.Readback"`). The `"PickingPass"` executor branch records `CopyTextureToBuffer(EntityId, TransferSrc, 0, 0, m_PickingReadbackBuffer, slot * 8, pickX, pickY, 1, 1)` + `CopyTextureToBuffer(PrimitiveId, TransferSrc, 0, 0, m_PickingReadbackBuffer, slot * 8 + 4, pickX, pickY, 1, 1)` after the four selection-ID sub-passes record, where `slot = frame.FrameIndex % m_Device->GetFramesInFlight()`; the copy is gated on `m_Device->IsOperational() && m_PickingReadbackBuffer->IsValid() && renderWorld.PickRequest.Pending` (the same signal that enabled `EnablePicking` upstream). The `(ColorAttachment → TransferSrc → ColorAttachment)` barrier pair wraps the two copies for both `EntityId` and `PrimitiveId`, mirroring the GRAPHICS-033D `MinimalDebugReadbackBuffer` pattern; the copies run *outside* the render pass via an idempotent `endActiveRenderPass()` latch (`renderPassEnded`) so the outer cleanup in the executor lambda is a no-op when the picking branch already ended the pass. The single-pixel region is delivered through an additive RHI seam: `ICommandContext::CopyTextureToBuffer` now takes four optional `(srcOffsetX, srcOffsetY, srcWidth, srcHeight)` parameters that default to `(0,0,0,0) = whole mip extent`, preserving the GRAPHICS-033D backbuffer-readback contract; the Vulkan backend interprets a non-zero `(width, height)` as a sub-rect (used here as `(pickX, pickY, 1, 1)`). The renderer publishes a per-frame `RenderGraphFrameStats::PickingReadbackCopyCount` counter that increments by one per frame in which the copy pair records — the CPU-observable contract Slice D.3 will key off. Contract tests landed: `PickingReadbackImportedFromRenderer` (compiled graph has `Picking.Readback` with `BufferImported=true`, `BufferHandles == renderer-supplied handle`, `(InitialState=TransferDst, FinalState=HostReadback)`, and the recipe-aware validator reports no errors); `PickingReadbackCopyRecordedWhenPickPending` (operational device + pending pick → `PickingReadbackCopyCount == 1` plus ≥ 2 `ColorAttachment→TransferSrc` and ≥ 2 `TransferSrc→ColorAttachment` barriers); `PickingReadbackCopySkippedWhenNotPending` (no pick → recipe drops `PickingPass`, counter stays at 0). Does not drain or publish (D.3 scope); does not touch the outline push constants (D.4 scope).

  - **Slice D.3 — `BeginFrame` drain + `SelectionSystem::PublishPickResult` / `PublishNoHit` wiring.** `NullRenderer` tracks the per-slot frame number and pending-request metadata (`m_PickingSlotIssuedFrame[slot]`, `m_PickingSlotRequest[slot]`, `m_PickingSlotInvalidated[slot]`) populated in `ExecuteFrame` when the D.2 copy is recorded. `BeginFrame(frame)` runs the drain *before* acquiring the new frame: for each slot whose `IssuedFrame` is older-than-or-equal-to `m_Device->GetCompletedFrameNumber()` (or `IDevice::HasFrameCompleted(frame)` equivalent), map the host-visible buffer at `slot * 8`, decode the two `uint32_t`s as `EntityId` and `EncodedSelectionId`, and route: `EntityId == 0` → `PublishNoHit()`; `Invalidated` → `PublishNoHit()`; otherwise `PublishPickResult(PickReadbackResult{.EncodedId = encoded, .StableEntityId = entity, .Hit = true})`. Clear the slot's metadata after the drain. Contract tests: (1) `PublishesPickResultForHitPixel` — extracted procedural triangle + `input.Pick.Pending = true` with pixel inside the triangle, after one frame's `BeginFrame()`-side drain the `SelectionSystem` receives `PublishPickResult` with `StableEntityId == StableEntityId(triangleEntity)`; (2) `PublishesNoHitForMissPixel` — same setup with pixel outside the triangle, `PublishNoHit` is called; (3) `PublishesNoHitForInvalidatedRequest` — `input.Pick.Pending` was set in frame N but cleared in frame N+1 before drain — `PublishNoHit` is called. Tests rely on `MockDevice` exposing the host-visible buffer contents via `BufferWrites` or a new `BufferContents` lookup so the test harness can seed the bytes the drain reads; if not, extend `MockDevice` minimally (in-scope for D.3).

  - **Slice D.4 — outline push-constant plumbing.** `Pass.Selection.Outline` exposes a `SelectionOutlinePushConstants` struct (outline color, width, selected-id list head + count) and `SelectionOutlinePass::Execute` pushes it before `Draw(3,1,0,0)`. The fragment shader's existing push block is wired to this struct (matching member offsets). `RenderWorld::Selection` already carries the selected stable id set; the recipe forwards a compact list (or a uniform-buffered tail when the count exceeds the push-constant limit; choose at implementation time after checking `RHI::DeviceLimits`). Contract test: `SelectionOutlinePushConstantsMatchRecipeInputs` asserts the recorded push constants match a seeded `RenderWorld.Selection` content. Independent of D.3 (does not need the drain); can land in parallel if the depending recipe wiring stays disjoint.

  Sub-slice ordering: D.1 lands first (lifecycle), then D.2 (recipe import + copy), then D.3 (drain + publish). D.4 (outline push constants) is independent of D.1..D.3 and can land at any point after Slice C; it is placed last to keep the picking-readback path contiguous in review. ## Goal
- Wire the existing `Pass.Selection.{EntityId,PointId,EdgeId,FaceId}` and `Pass.Selection.Outline` classes into the renderer executor under the default recipe, allocate the host-visible `Picking.Readback` buffer, implement the readback drain on `BeginFrame()` after the issuing frame's fences complete, and route `SelectionSystem::PublishPickResult` / `PublishNoHit` per `GRAPHICS-012`/`012Q`.

## Non-goals
- No transparent / special-forward picking eligibility (`GRAPHICS-025` planning).
- No editor selection policy (runtime/editor owns it per `GRAPHICS-012Q`).
- No transform-gizmo hit testing (that lives in `RUNTIME-014` `GizmoInteraction`).
- No ECS mutation from graphics.
